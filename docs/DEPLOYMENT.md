# Deployment

## Why

EmbedIQ Cloud ships safe defaults for local development, but a public-facing deployment needs a
hardened configuration. This page explains the difference and how to run the production profile.

## Development vs production

| | Development | Production |
| --- | --- | --- |
| Command | `bash scripts/dev.sh` or `docker compose up -d` | `docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d` |
| Broker | Anonymous; plain `1883` and TLS `8883` | TLS/mTLS only on `8883`; no plain `1883` |
| Device auth | Off (anonymous) | Enforced via the go-auth plugin → `/internal` |
| `INTERNAL_API_KEY` | Optional | **Required** (the app refuses to start without it) |

`scripts/dev.sh` runs the API and admin panel with SQLite and no broker — for previewing the UI,
not for serving devices.

## Production profile

The production overlay `docker-compose.prod.yml` layers on top of the base file. Run both:

```sh
bash install.sh
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d --build
```

What the overlay changes:

- **Broker is TLS-only.** `config/mosquitto/mosquitto.prod.conf` publishes only `8883` with
  `require_certificate true`; the plain `1883` listener is not exposed. Anonymous access is off.
- **Devices authenticate through the API.** The Mosquitto go-auth plugin calls
  `/internal/auth` (and `/internal/acl`) on the API to validate every client.
- **`EMBEDIQ_PROFILE=prod`** is set on the API container, which makes the shared secret mandatory.
- **`INTERNAL_API_KEY` is required.** The API calls `assert_internal_key_in_prod` at startup and
  **refuses to boot** if the profile is `prod` and the key is empty. The broker presents this
  secret to the internal endpoints; the endpoints reject any unkeyed call.

## Certificates

`install.sh` (or `scripts/gen-certs.sh` directly) generates the CA and server certificates into
`./certs`, mounted read-only into the broker. The script is idempotent — it reuses existing
certificates and never overwrites them silently.

## Reverse proxy and exposure

The API speaks plain HTTP. In production, terminate TLS at a reverse proxy (for example Caddy,
nginx, or Traefik) in front of port `8080`.

- Do **not** expose `8080` directly to the public internet.
- In particular, the `/internal/*` endpoints are an allow/deny oracle; keep them unreachable
  from outside the host. The shared-secret guard is defence in depth, not a substitute for not
  publishing the port.

## Caveats

- Device-side certificate rotation (EST) is performed by firmware, not the cloud. The cloud
  tracks expiry and can trigger a rotation; the on-device exchange is out of scope here.
- `INTERNAL_API_KEY`, the admin password hash, and all service secrets live in `.env`
  (generated by `install.sh`) and are never committed.
- The lighter `docker-compose.lite.yml` profile omits hawkBit and MediaMTX for ~1 GB hosts; OTA
  and streaming are unavailable in that profile.
